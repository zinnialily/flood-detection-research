The Problem

Street cameras and social media produce a flood of images during disasters. Manual review is too slow. Automated classifiers are fast — but they make dangerous mistakes.

🌊

Rapid flood detection

Street-level cameras, dashcams, and social media images contain early flood signals — but manual review is too slow for emergency dispatch. An automated filter could identify flooded scenes within seconds.

🏊

The confounder problem

Swimming pools, wet roads, and rain-obscured scenes look visually similar to flooded streets. A naive classifier trained without these hard negatives triggers false alarms — and false dispatch calls cost lives and resources.

🎯

High recall over precision

In emergency response, missing a real flood is far more costly than a false alarm. This project explicitly targets ≥ 95% recall — catching more real floods is the primary design constraint, not minimising false positives.

Phase 1 · CIBB 2026

Original Conference Paper

Built a binary flood classifier using EfficientNetB0 and ResNet50 with two-phase progressive fine-tuning. Applied hard negative mining (HNM) to EfficientNetB0 after discovering that swimming pools caused a 20% false positive rate. Results looked impressive.

Model Accuracy Recall F1 AUC Pool FP
EfficientNetB0 85.46% 82.22% 86.33% 93.59% 20.0%
ResNet50 94.15% 94.60% 94.75% 98.15% 13.3%
EfficientNetB0 + HNM 98.76% 99.37% 98.89% 99.93% 0.0%
⚠️ These results were produced with a preprocessing bug — see Phase 2. The apparent improvement conflates bug effects, extra training time, and actual HNM contribution. Do not interpret at face value.

Phase 2 · Bug Discovery

Three Critical Problems Found

🐛 Bug 1: Double-rescaling — transfer learning was non-functional

The original code used ImageDataGenerator(rescale=1./255) AND the backbone's built-in preprocessing (which also divides by 255). This compressed pixel values to ≈[0, 0.004] — 255× too small. The frozen pretrained ImageNet features produced near-zero activations through ReLU nonlinearities. Transfer learning was essentially broken. The "85.46% baseline" is really the performance of a partially random network, not a fine-tuned model.

❌ Old code (train_efficientnet.py)

train_aug = ImageDataGenerator(
    rescale=1.0 / 255,  # ← BUG
    rotation_range=15,
    horizontal_flip=True,
    ...
)

base_model = EfficientNetB0(
    weights='imagenet',
    include_top=False,
    include_preprocessing=True  # ← BUG
    # built-in layers ALSO divide by 255
)

✅ Fixed code (scripts/utils.py)

train_aug = ImageDataGenerator(
    preprocessing_function=preprocess_input,
    rotation_range=15,
    horizontal_flip=True,
    ...
)

base_model = EfficientNetB0(
    weights='imagenet',
    include_top=False,
    include_preprocessing=False  # ← correct
    # preprocessing handled by generator only
)

🔬 Issue 2: Confounded architectural comparison

HNM was only applied to EfficientNetB0, not ResNet50. The comparison was therefore EfficientNetB0+HNM+extra-training vs ResNet50-without-either. Any conclusion about which architecture is "better with HNM" was completely unsupported — the two variables (architecture and training) were never separated. Fix: add ResNet50+HNM to the pipeline.

📊 Issue 3: Decision threshold selected on the test set

The reported optimal threshold (0.85) was chosen by sweeping on test data — this is circular validation. Selecting a threshold by the metric it will be evaluated on guarantees overfitting to that specific set. The current pipeline selects all thresholds on the validation set only, then reports test performance at that threshold exactly once.

Methodologically Rigorous Pipeline

Complete rewrite with six key improvements — pipeline is done, awaiting GPU quota to run:

→

Preprocessing bug fixed

Explicit preprocessing_function, include_preprocessing=False, assertion cell before every training loop

→

ResNet50 + HNM added

Unified --arch flag enables fair architectural comparison via identical training protocols

→

Extended-training control

--no_injection flag: same epoch budget without HNM data — isolates the actual HNM contribution from the benefit of more training time

→

PR-AUC as primary metric

ROC-AUC is misleadingly optimistic under class imbalance in disaster detection. Precision-recall AUC measures the trade-off that matters operationally.

→

Bootstrap CIs + McNemar test

5-seed runs with mean ± std. No single-run results on 564 images without confidence intervals. McNemar test for paired model comparison.

→

Severity-stratified recall

Major/Moderate/Minor flood recall reported separately — connects classification to operational emergency response thresholds (geoscience contribution).

Pipeline steps (simplified for mobile)

①

Baseline training

EfficientNetB0 + ResNet50, ImageNet weights, 2-phase fine-tuning ×5 seeds

②

Confounder analysis

Predict on train/non_flood/ — rank categories by false positive rate

③

Hard negative mining

Mine training set only (tau=0.3), augment 5×, assert zero val/test leakage

④

Retrain configurations

HNM retrain + extended-training control (same epochs, no injection)

⑤

Threshold selection

Sweep on val set only — test set never used for tuning

⑥

Final evaluation

PR-AUC (primary), severity-stratified recall, bootstrap CIs, McNemar test

Data integrity

Hard negatives mined only from training set. Assertion cell halts notebook if any file appears in both train and val/test.

Threshold hygiene

All threshold selection on validation set. Test set used exactly once per model configuration — never for tuning.

Ablation control

Extended-training control runs the same total epochs without HNM injection — isolates whether the improvement is from data or just training time.

Preliminary Results

⚠️ Preprocessing bug present in these results

These numbers come from the CIBB 2026 code which had a double-rescaling bug that rendered transfer learning non-functional. The baseline figures (85.46%, 94.15%) are likely underestimates of what correct preprocessing will achieve. The HNM jump (85% → 99%) conflates at least three effects: (1) preprocessing fix, (2) additional training epochs, and (3) actual HNM data contribution. These will be replaced with corrected numbers after GPU runs complete. PR-AUC, bootstrap CIs, McNemar test results, and severity-stratified recall will be added then.
